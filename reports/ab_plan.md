# Day 5 A/B Test Plan - Fulfillment SLA Upgrade

## 实验目标

验证一个真实可执行的履约实验：对首单用户订单提供更快的履约 SLA，是否能减少晚到和差评，并最终提升 `30 天复购率`。

这次不做“评价文案微调”，而选择 `履约 SLA`，原因是前几天的分析已经显示：晚于承诺时间会显著拉高差评，而差评用户后续复购更弱。相比之下，先修复履约体验更接近高杠杆动作。

## 假设

- `H1`：更快的履约 SLA 会降低晚到率，尤其是 `4 天以上晚到` 的严重异常占比。
- `H2`：更好的履约体验会降低差评率和客服压力。
- `H3`：首单体验改善后，用户 `30 天复购率` 会提升。

## 实验方案

### 实验名称

`First-order SLA Upgrade`

### 实验对象

- 用户口径：`首个有效订单用户`
- 触发时点：支付成功后立即分流
- 资格条件：
  - 有效订单
  - 卖家 / 线路支持至少两种履约方案
  - 非超大件、非特殊运输限制商品

### Treatment

对实验组订单执行一套更快的履约策略：

- 优先进入更早的打包 / 发货波次
- 在可选时优先分配更快承运商
- 对高风险晚到订单提前触发异常处理

### Control

保持当前默认履约策略。

### 随机化与分层

- 随机化单位：`customer_unique_id` 对应的首单用户
- 分流比例：`50 / 50`
- 分层建议：
  - `promised_sla_bucket`：`<7d`、`7-14d`、`14-21d`、`21d+`
  - `destination_region`：至少按大区分层
  - `basket_value_bucket`：`<100` vs `100+`

这样可以减少“长链路订单更多落在某一组”带来的偏差。

## 指标设计

### 北极星主指标

- `30 天复购率`
  - 定义：首单支付成功后 30 天内，用户是否再次产生新的有效订单
  - 原因：这是最接近业务价值的结果指标

### 过程指标

- `晚到率`
  - 定义：`order_delivered_customer_date > order_estimated_delivery_date`
- `4 天以上晚到率`
  - 定义：`delivery_delay_vs_estimate_days > 4`
- `差评率`
  - 定义：`review_score <= 2`

过程指标的作用不是替代主指标，而是更早验证机制有没有按预期生效。

### 护栏指标

- `退款率`
- `取消率`
- `差评率`
- `客服量 / 每千单`
- `物流成本 / 每单`

其中 `差评率` 既是过程指标，也是护栏，因为如果为了追求更快履约造成体验反噬，它必须第一时间被拦住。

## 历史基线

基于 Olist 历史数据中的首单用户：

- `30 天复购率`：`1.62%`
- `60 天复购率`：`1.98%`
- `晚到率`：`8.01%`
- `4 天以上晚到率`：`5.05%`
- `差评率`：`14.26%`
- `首单用户日均流量`：约 `155`

这些基线说明两件事：

1. `30 天复购` 非常低频，单独做 power 会很慢。
2. 晚到和差评更适合作为阶段性读数指标。

## MDE 与样本量

### 主指标 MDE

我建议把 `30 天复购率 +0.50pp` 作为最小值得检测的业务提升：

- 从 `1.62%` 提升到 `2.12%`
- 相对提升约 `30.8%`

按双侧检验、`alpha = 0.05`、`power = 0.80` 估算：

- 每组样本量约 `11,549`
- 总样本量约 `23,098`

如果按 Olist 历史的首单流量 `155 / 天` 估算：

- 招募时长约 `149` 天
- 再加 `30` 天结果观察窗
- 全量主指标读数约需 `179` 天

这已经说明：`30 天复购` 可以做北极星指标，但不适合拿来做唯一的短期放量依据。

### 过程指标 MDE

建议把 `晚到率下降 1.5pp` 作为阶段性放量门槛：

- 基线 `8.01%`
- 目标 `6.51%`
- 每组约 `5,574`
- 总样本约 `11,148`
- 按历史流量，招募约 `72` 天

这个周期仍然不算短，但明显比复购主指标更可操作。

## 周期与放量计划

### 建议节奏

1. `A/A` 校验 1 周  
   验证埋点、分流和履约分配逻辑没有偏。

2. `10%` 小流量 3 到 5 天  
   先确认没有异常的取消、差评、客服暴涨。

3. `50 / 50` 正式实验  
   进入稳定收样阶段。

4. 第一阶段读数  
   当达到过程指标样本量后，先看：
   - 晚到率是否显著下降
   - 差评 / 客服 / 退款是否没有恶化

5. 第二阶段读数  
   保留 holdout，继续累计到 `30 天复购` 所需样本量，再做最终商业判断。

## 判定标准

### 可以继续放量

- 晚到率显著下降
- `4 天以上晚到率` 同方向下降
- 差评率没有显著恶化
- 客服量和退款率未超过预设阈值

### 不建议放量

- 晚到率改善很小或没有改善
- 物流成本明显上升，吞掉潜在收益
- 差评率、退款率、客服量任一明显恶化

## 分析方法

- 采用 `Intent-to-Treat` 作为主分析口径
- 比例指标使用双样本比例检验
- 同时做分层结果复盘：
  - 不同 `promised_sla_bucket`
  - 不同地区
  - 不同客单价区间

如果某个 bucket 受益显著，而其他 bucket 几乎无变化，可以进一步转成更精细的定向策略，而不是全量铺开。

## 主要风险

1. 处理组成本过高  
   更快的承运商和更高优先级会提高单单成本，所以必须把 `物流成本 / 单` 作为硬护栏。

2. 供给侧污染  
   如果处理组挤占了控制组的仓配资源，可能低估真实效果，也会污染对照组。

3. 季节性 / 大促干扰  
   大促周、节假日和促销月会同时影响履约与复购，实验排期要避开极端高峰，或至少在分析时单独标注。

4. 只改善过程，不改善业务  
   晚到率下降不一定自动转成复购提升，所以一定要保留长期 holdout，不能只看过程指标就宣布成功。

## 结论

如果把这份方案放在真实工作里，我会把它定义成一个 `两阶段实验`：

- 短期看机制：晚到率、严重晚到率、差评率、客服量
- 长期看业务：`30 天复购率`

在 Olist 这种低频场景里，这比“只看一次复购 uplift”更现实，也更符合增长和履约团队的协作方式。
